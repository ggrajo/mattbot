MattBot Phase 1 recap (everything decided/discussed in this chat about the project)
Note: This is NOT the technical plan chapter outline. This is the consolidated decisions, constraints, requirements, and open items we already discussed.

============================================================
1) What Phase 1 is (product promise)
============================================================
- Product: MattBot
- Phase 1 promise (core value):
  - A phone concierge MVP that answers calls when the user is busy or unreachable
  - Screens callers with a short conversation
  - Sends the user a clear summary and transcript
  - Includes strong privacy and security controls as MVP scope (not “later”)

- Phase 1 deliverable option selected:
  - Option C: Mobile app + backend + inbound AI calling + call forwarding mode + live handoff included

- Phase 1 is intentionally narrow (single main channel first):
  - Phone calls only as the primary channel for Phase 1 (inbound focus)

============================================================
2) Who Phase 1 is for (market + constraints)
============================================================
- Target users for Phase 1: personal users only
- Estimated users in first ~3 months after launch: 50 to 100 or less
- Phase 1 launch country: US only
- Supported languages in Phase 1: English only
- Timezones in Phase 1: auto-detect per user
- Platforms in Phase 1: iOS + Android (both)
- Multi-device behavior:
  - A single user account can be logged in on multiple devices at the same time: YES

============================================================
3) Core call handling modes (inbound)
============================================================
Phase 1 supports inbound call handling via two modes (users can enable either or both):

A) Dedicated AI number mode
- User is assigned an AI-managed phone number
- Callers can dial this AI number directly
- User controls who is allowed to reach this number (example rule types: everyone, contacts only, VIP only)
- Provisioning approach: auto-provision at onboarding (assign the number during setup)

B) Conditional call forwarding fallback mode (required in Phase 1)
- User keeps their normal phone number
- When the user is busy or unreachable, calls are forwarded to the AI
- Key promise: AI only answers forwarded calls when the user cannot answer (not every call)

Forwarding triggers (final selection not explicitly finalized in chat):
- Forwarding is required in Phase 1
- Trigger set to “best approach” (candidates discussed):
  - busy only
  - unreachable only
  - busy + unreachable

Carrier support constraints:
- “Decide what’s best” was requested (no final list set in chat)

Forwarding misconfiguration risk:
- The product must detect and warn about unconditional forwarding (risk: AI answers every call)

============================================================
4) Live handoff (required in Phase 1 deliverable)
============================================================
- Live handoff is included (Option C chosen)
- Live handoff concept:
  - AI answers and screens first
  - Under certain conditions, user can “take over” the call live
- Live handoff condition settings discussed (from the master plan):
  - Off
  - VIP only
  - Urgent only
  - VIP + Urgent
- The “loop” risk (explicit question raised):
  - If handoff destination is the user’s real phone number and the user is still busy/unreachable, this could cause repeated forwarding/loops
  - Best approach was requested, but not finalized in chat as a concrete rule set
  - This remains a known risk item to design around in the technical plan

Handoff destination (not finalized in chat):
- Candidates discussed:
  - User’s real phone number
  - In-app VoIP call
  - Choice per user

Handoff failure behavior (required):
- If handoff attempt fails, conversation must continue (no dead air)

============================================================
5) Screening conversation (what the AI does on the call)
============================================================
- Goal of screening:
  - Collect only what the user needs to decide what to do next
- Core capture targets:
  - Who is calling
  - Why they are calling
  - How urgent it is
  - How to reach them back
- If caller refuses to answer screening questions:
  - fallback to simple message-taking so the call still produces usable output

Temperament and identity boundaries:
- The assistant must present itself as an assistant (not pretend to be the user)
- Avoid harassment/insults/threats
- Avoid sharing sensitive info
- Avoid strong claims without evidence
- Focus on capturing message details, not “solving everything”

============================================================
6) Post-call artifacts (what the user receives)
============================================================
After each call, the user should have a call record that includes:
- Concise summary (fast to read)
- Full transcript (for detail)
- Labels (example categories discussed: urgency, spam, sales, VIP, normal, unknown)
- Label reasons (for transparency)
- Optional recording link if recording is enabled

Failure tolerance requirement:
- If any AI component fails, the system must still create a call record and clearly mark missing parts

16) Security baseline (must be defined because personal data)
============================================================
Security expectations explicitly treated as mandatory MVP scope:
- Data protection in transit and at rest
- Strong access controls to prevent cross-user data exposure
- Biometric protection for call content (transcripts, recordings, and sensitive fields per policy)
- Clear retention and deletion behavior
- Audit logs for sensitive actions (device changes, deletions, etc.)
- Abuse prevention controls (rate limits, blocking)

Encryption at rest requirement:
- Everything sensitive (server-side + any cached client-side content)

Local storage rules (mobile):
- If summaries/transcripts are cached:
  - store encrypted on device
  - clear cache on logout, device revoke, delete-all, account deletion
  - enforce biometric gating before display, based on selected policy

Network + DB security (raised as a gap to define):
- You explicitly asked to ensure network security and DB security are fully defined because of personal data
- Concrete implementation choices were not fully enumerated in chat as final decisions
- This remains a required section to lock down in the technical plan

Abuse prevention (requested to decide, not finalized in chat):
- rate limits per user, per IP, per phone number, etc.
